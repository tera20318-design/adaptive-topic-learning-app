# Report Template

## Rule

The final report is a reader-facing finished document.
It is not a dump of internal pipeline state.

## Required Semantic Sections

Every report must cover these semantic slots, though headings may be adapted to reader-facing wording:

1. Direct answer
2. Scope and exclusions
3. Core explanation
4. Decision-relevant analysis
5. Main options / categories / mechanisms
6. Risks and failure modes
7. Evidence-backed findings
8. Reader decision layer
9. Checklist or decision table
10. Uncertainty and next research
11. Sources

## Writing Rules

- lead with the answer, not the process
- separate facts, inferences, and advice clearly
- keep scoped limitations visible near the top and near uncertainty sections
- use decision-useful tables where the domain adapter requires them
- keep the sources section auditable and consistent with the citation ledger

## Prohibited Patterns

- using internal pipeline headings as section titles
- pasting an evidence matrix into the body as if it were a report
- treating source count as proof of quality
- reducing the report to a claim-ledger summary

## Examples Of Prohibited Internal Headings

- `role structure`
- `upstream downstream`
- `contradiction notes`
- `domain risk map`
- `claim audit`
- `evidence matrix`
